**Here is a precise table of the binary structure of a Solana DataShred** that contains a Pump.fun transaction (e.g. a `buy_v2`).

Solana DataShreds are fixed-size packets (~1228 bytes) optimized for UDP/MTU. The Pump.fun payload (24 bytes) lives deep inside the serialized entries/transactions.

### DataShred Overall Binary Layout (current versions)

| Offset (from start of shred) | Size     | Field                          | Description / Notes |
|------------------------------|----------|--------------------------------|---------------------|
| 0x00 – 0x3F                 | 64 B     | Leader Signature               | Ed25519 signature (or Merkle root in v3) |
| 0x40                         | 1 B      | Variant                        | Shred type + auth scheme (data shred range) |
| 0x41 – 0x48                 | 8 B      | Slot                           | u64 little-endian |
| 0x49 – 0x4C                 | 4 B      | Shred Index                    | u32 LE (position in FEC set) |
| 0x4D – 0x4E                 | 2 B      | Version                        | u16 LE |
| 0x4F – 0x52                 | 4 B      | FEC Set Index                  | u32 LE |
| 0x53 – ~0x57                | ~5 B     | Data-specific header (v1/v2)   | Parent offset (u16), flags, size, etc. |
| ~0x58+                       | up to ~1003–1100 B | **Payload** (Entry Batch)     | **This is where the Pump.fun data lives** |

(The exact payload start can shift slightly with Merkle proofs in newer auth schemes — parsers usually skip the known header size.)

### Inside the Payload → Entry Batch
- **u64** (or compact) = Number of Entries in this batch
- Then concatenated **serialized Entries**

### Inside one Entry
| Field                  | Size / Type          | Notes |
|------------------------|----------------------|-------|
| num_hashes             | u64                  | Usually 1 |
| hashes                 | 32 B each            | PoH proof |
| num_transactions       | CompactU16           | — |
| Transactions           | Variable             | List of serialized txs (including your Pump.fun one) |

### Inside one Transaction (Versioned v0, most common)
| Field                        | Size / Type              | Notes |
|------------------------------|--------------------------|-------|
| Version marker               | 1 B (0x80 \| version)    | — |
| Signatures                   | CompactU16 + 64 B each   | Usually 1 signature |
| Message header               | 3 B                      | (req_sigs, readonly_signed, readonly_unsigned) |
| Account keys                 | CompactU16 + 32 B each   | Pump program is one key |
| Recent blockhash             | 32 B                     | — |
| Instructions                 | CompactU16 + list        | **Pump buy_v2 instruction here** |
| (Optional LUTs)              | —                        | — |

### Inside the Pump.fun `buy_v2` CompiledInstruction
| Field                    | Size / Type      | Content |
|--------------------------|------------------|---------|
| program_id_index         | 1 B              | Index of Pump program in account keys |
| accounts                 | CompactU16 + u8s | Always **27** accounts (fixed order) |
| data_length              | CompactU16       | Usually `0x18` (24) |
| **data (payload)**       | **24 B**         | `b8 17 ee 61 67 c7 d3 3d` + u64 amount (LE) + u64 max_sol_cost (LE) |

This is the exact spot where your Pump.fun buy payload appears in the raw binary stream of the shred.

The structure is stable, but header details can have minor version differences (v1/v2/v3 Merkle). For production parsing, use libraries like the ones in orbitflare/shredstream-decode-example or Solana’s own `shred` crate.

**Yes — the 27 accounts for `buy_v2` are always exactly the same.**

Pump.fun made `buy_v2` (and `sell_v2`/`buy_exact_quote_in_v2`) deliberately **fixed and mandatory** so every transaction uses the identical account order and flags. This is explicitly stated in their official docs:

> “All accounts are mandatory and are passed in the same order for all kinds of coins.”  
> “No optional accounts — unified interface for both SOL-paired and non-SOL-paired coins.”

### What this means in practice
- The **order** of the 27 accounts **never changes**.
- The **isWritable / isSigner** flags **never change**.
- You always pass **all 27**, even if some are dummy/reused for SOL coins.

Only the **actual Pubkey values** change per transaction (e.g. different mint, different bonding curve PDA, different user ATA, etc.).

### The fixed 27-account list (in exact order you must provide)

| #  | Account name                              | Usually mutable? | Signer? | Notes / How it changes |

### Why they did this
It makes bot writing, MEV searching, and parsing **much more reliable** — you can hard-code the account order and just swap in the correct PDAs/ATAs for each coin.

**Legacy `buy`** (the old one) had fewer accounts (~14–16) and some were optional. That’s why most new bots now use `buy_v2`.
